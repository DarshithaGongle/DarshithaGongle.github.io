01 / The problem

A moderation problem
framed wrong

Digital platforms have a toxicity problem, but the existing solutions treat it as a content removal problem rather than a communication quality problem. Instagram, Facebook, and most platforms rely on rule-based systems or simple ML classifiers that either miss nuanced harmful language or delete flagged content without explanation.

The real user pain is not just that toxic content exists. It is that users have no feedback loop — no understanding of why their language is harmful, and no tools to self-correct. Neutral Note was built to fill that gap: detect toxic language in real time and immediately offer the user a neutral alternative, turning moderation into education.

02 / The product decision

Why rephrase.
Not flag. Not delete.

This was a deliberate product philosophy decision, not a technical default. Flagging or deleting content is punitive without being constructive — it tells a user their message is unacceptable but leaves them no better equipped to communicate differently.

When toxic content is detected, the system presents the user with three things simultaneously: the original text, the toxicity category it triggered, and a rephrased neutral alternative. The rephrased version preserves the original communicative intent while removing the harmful tone. The user's idea is not erased — it is improved.

Concrete example
Input: "You are alcoholic" → Category: Insult → Rephrased: "It seems like you're struggling with alcohol use."
The meaning shifts from accusatory to empathetic while carrying the same underlying observation.

This approach also has practical advantages in professional contexts — outright deletion of a message in a workflow like an email draft or customer service reply would be disruptive. A neutral alternative keeps communication moving while removing the harm.

03 / The architecture

Why unbiased Detoxify.
Why GPT-4.

Detoxify offers three model variants: original, unbiased, and multilingual. The unbiased variant was chosen deliberately — not as a default — because toxicity is rarely about individual words. The same word can be neutral or harmful depending on sentence structure, target, and intent. The unbiased model evaluates surrounding context rather than pattern-matching on surface vocabulary.

Option considered
Detoxify original
Faster, lighter. But trained on data that showed documented bias against African American English dialect — flagging non-toxic AAE text as harmful.
Chosen
Detoxify unbiased
Built on BERT/RoBERTa with self-attention across full sentence context. Specifically trained to reduce identity-based false positives. The right call for responsible AI.
Option considered
Custom classification model
Would require large labeled dataset, training time, and ongoing evaluation. Detoxify already offered a pre-trained, validated transformer architecture.
Chosen for rephrasing
GPT-4 Turbo via API
4096-token context window versus Detoxify's 512-token BERT limit. Capable of understanding not just what the input says but why it is harmful, then restructuring it naturally.
04 / The bias finding

Catching what the
model got wrong

The standard Detoxify original model has a documented problem: it was trained on data that underrepresented African American English (AAE) dialect, causing it to flag non-toxic AAE text as harmful at a higher rate than equivalent Standard American English text. This is a form of algorithmic bias — the model's definition of "toxic" was shaped by who generated the training data, not by what is actually harmful.

Key insight
This was not just a technical bug. It was a product decision with social consequences. A moderation tool that disproportionately flags one dialect of English is not a neutral tool — it is an enforcement mechanism with a bias built in. Choosing the unbiased model was the product-correct decision, not just the technically-correct one.

The unbiased model evaluates text across nine identity labels — male, female, homosexual/gay/lesbian, Christian, Jewish, Muslim, Black, White, and psychiatric or mental illness — allowing it to detect not just overt aggression but targeted identity-based attacks while reducing false positives across dialects and communities.

The classification threshold was set at 0.65 across all seven toxicity categories. This was calibrated to balance precision against recall — high enough to avoid flagging ambiguous language, low enough to catch genuine harm. A threshold of 1.0 would catch nothing. A threshold of 0.1 would flag everything. 0.65 is a deliberate product decision, not a default.

05 / The pipeline

Input to output,
every step

01
Authentication
User registers via Django auth. Passwords encoded with PyJWT, stored in MariaDB via XAMPP. Session managed through Django's built-in auth system.
02
Text input via AJAX
Chatbot-style input field. AJAX handles form submission without page reloads. Text sent asynchronously to Django backend through URL routing.
03
Sentence segmentation
Input text is split on ". " into individual sentences. Each sentence is analyzed independently to preserve per-sentence toxicity granularity.
04
Detoxify classification
Each sentence passed to Detoxify unbiased model. Tokenized into subword units, converted to semantic embeddings, processed through stacked self-attention layers. Output: probability scores for 7 toxicity categories. Threshold: 0.65.
05
GPT-4 rephrasing (toxic only)
If any category score exceeds 0.65, sentence sent to GPT-4 Turbo via OpenAI API. Prompt instructs GPT-4 to rewrite as neutral while preserving original communicative intent. Response extracted from choices[0].message.content.
06
Output rendered
Each sentence mapped to: original text, toxicity category label, and rephrased alternative. All three displayed side-by-side in chatbot interface via displayRephrasedText().

08 / What comes next

The V2 roadmap

Ensemble learning
Bagging, boosting, or stacking multiple models to aggregate predictions and reduce the impact of individual model weaknesses. Directly addresses current false positive and false negative rates.
Domain-specific fine-tuning
Fine-tune on journalism, healthcare, or corporate communication datasets. Language that is neutral in isolation can be harmful in a specific professional context — general training data misses this entirely.
Real-time detection
Flag language as the user types rather than after submission. Shifts from reactive moderation to proactive prevention — reducing harmful content before it is even finalized.
Multi-language support
Current system is English-only. Extending detection and rephrasing to other languages — with culturally-adjusted thresholds — is the highest-priority expansion to reach global users.
Social media API integration
Browser extension or platform API integration moves Neutral Note from a standalone tool to an embedded real-time moderator operating at the point of content creation on Twitter, Instagram, and Facebook.
User-adjustable thresholds
Allow professionals to tune detection thresholds and rephrasing styles to their domain. Appropriate language in a legal document is not the same as in a social media post.
01 / The problem

70 million people.
No real-time bridge.

Over 70 million deaf individuals worldwide rely on sign languages like ASL and ISL to communicate. The vast majority of hearing people have no sign language proficiency, and professional interpreters are expensive, scarce, and unavailable in most everyday situations — a deaf student in a classroom, a patient describing symptoms to a doctor, a job applicant in an interview.

Existing tools are almost universally unidirectional. They either convert sign to text, or speech to sign — never both. They rely on static datasets, sensor-based hardware like gloves, or low-quality animations that break natural conversational flow. SignVoice was built to close that gap with a web-based, bidirectional, real-time platform supporting both ASL and ISL — no hardware required.

02 / The product decision

Bidirectional by design.
Not by accident.

Most existing tools are translation pipelines in one direction. The fundamental design decision behind SignVoice was to build genuine two-way communication — which required two completely separate technical subsystems, not one system run in reverse.

Sign to Speech
For deaf and hard-of-hearing users
Webcam input → MediaPipe landmarks → CNN classification → word assembly via Enchant → Microsoft TTS (English) or Google Translate + gTTS (Hindi).
Speech to Sign
For hearing users
Microphone input → Webkit Speech Recognition → text tokenization → Blender 3D avatar animation library → animated sign playback with speed controls.
Why both directions matter
A deaf student cannot participate in a classroom where the teacher can sign to them but they cannot sign back and be understood. Real conversation requires both directions. Building only one would have replicated the very exclusion the system was trying to solve.
03 / The architecture

MediaPipe. OpenCV. CNN.
Three layers, one purpose.

The sign recognition pipeline is built on three technologies working in sequence — each handling a distinct part of the problem.

MediaPipe processes the live webcam feed and extracts 21 key landmarks per hand, covering all fingertips, knuckle joints, and palm orientation. Instead of asking a neural network to interpret a raw pixel image, the system feeds it a clean geometric representation of hand shape — exact 3D coordinates of every relevant point.

OpenCV converts each frame from raw color to grayscale then to binary — a high-contrast representation that strips away background noise and lighting variation. The CNN does not need to learn that the same gesture looks different under fluorescent versus natural light. That problem is solved before the model ever sees the image.

The CNN (TensorFlow and Keras) then classifies the preprocessed binary image against its training dataset. Trained on labeled ASL and ISL gestures with data augmentation — rotation, flipping, scaling, brightness adjustment — to generalize across different users, cameras, and environments. Achieved 94.2% accuracy on static alphabet signs.

04 / The technical insight

Why geometry beats
the CNN alone

The CNN achieves 94% accuracy, but some ASL signs are so visually similar that a pure image classifier struggles to distinguish them reliably. The signs for A, S, and E all involve a closed fist — the difference is only in thumb positioning. M, N, and T are nearly identical at a glance.

The solution is a two-stage classification system. First, the CNN groups signs into 8 broad clusters (ch1 == 0 through ch1 == 7) based on overall hand shape. Then, within each cluster, geometric rules using landmark distances and positional relationships make the final determination.

Example — disambiguation in action
Signs A, S, E, M, N, T all land in group 0. The CNN says "closed fist." Then landmark geometry takes over: if pts[4][0] is greater than pts[6][0] and pts[4][0] is greater than pts[10][0] and greater than pts[14][0] — thumb position relative to finger joints — the system classifies as A. Distance between pts[12] and pts[4] distinguishes C from O. Distance between pts[8] and pts[12] separates G from H. The geometry knows what the pixel classifier cannot.
05 / The inclusivity decision

ASL and ISL.
Not interchangeable.

Supporting both sign languages was not a feature addition — it was an inclusivity requirement. A system that only supported ASL would exclude the Indian deaf community the project was built to serve. A system that only supported ISL would limit international reach and compatibility with the larger global body of sign language research.

Critically, ASL and ISL are not the same with different labels. While both use the A-to-Z alphabet, the hand shapes, finger orientations, and positional conventions for individual letters differ meaningfully between them. The same letter is not the same gesture.

This difference runs through every layer of the system — separate training datasets, separate Blender-rendered 3D animation libraries, separate SVG dictionary icons, separate quiz banks in the learning module, and a real-time toggle that switches between them across all four modules simultaneously.

06 / The pipeline

Sign to Speech,
every step

01
Webcam capture
Live video feed captured in browser. Each frame passed to the processing pipeline in real time.
02
MediaPipe landmark extraction
21 hand landmarks extracted per frame — fingertips, knuckle joints, palm orientation — giving the system a 3D geometric map of hand shape.
03
OpenCV preprocessing
Frame converted to grayscale then binary. Background noise, lighting variation, and irrelevant visual data stripped before classification.
04
CNN classification with confidence threshold
Binary image classified against training dataset. If confidence falls below threshold, user is prompted to re-sign rather than propagating a low-confidence character into the word buffer.
05
Geometric disambiguation
Within each of the 8 CNN groups, landmark distance and positional rules make the final character determination for visually similar signs.
06
Word assembly via Enchant
Recognized characters accumulated in a letter buffer. Python Enchant library generates word completion suggestions in a horizontal suggestion bar as letters build up.
07
Speech synthesis
Completed sentence passed to Microsoft TTS API for English output, or Google Translate API plus gTTS for Hindi — giving the system genuine bilingual spoken output.

# Stage 1: CNN outputs one of 8 broad hand-shape groups (ch1 == 0..7) # Stage 2: geometric rules use landmark distances to make final call # This is why the system beats a pure pixel classifier on similar signs def predict(self, test_image): white = test_image.reshape(1, 400, 400, 3) prob = np.array(self.model.predict(white)[0], dtype='float32') ch1 = np.argmax(prob) # top group from CNN (0-7) # Group 0 = closed fist family: A, E, M, N, S, T # Landmark geometry disambiguates — thumb position relative to fingers if ch1 == 0: ch1 = 'S' # default if self.pts[4][0] < self.pts[6][0]: # thumb inside fingers ch1 = 'A' if self.pts[4][1] > self.pts[8][1]: # all fingers curled under ch1 = 'E' if self.pts[4][0] > self.pts[10][0]: # thumb crosses middle ch1 = 'M' # Group 2 = rounded hand: C or O # Distance between thumb tip (pt4) and middle finger (pt12) is the key if ch1 == 2: ch1 = 'C' if self.distance(self.pts[12], self.pts[4]) > 42 else 'O' # Group 3 = extended index family: G or H # Distance between index (pt8) and middle (pt12) fingertips if ch1 == 3: ch1 = 'G' if self.distance(self.pts[8], self.pts[12]) > 72 else 'H'
09 / What comes next

The V2 roadmap

LSTM dynamic gesture recognition
Current CNN classifies static snapshots. V2 integrates LSTM/RNN sequence models to process sign input as a time series — recognizing full words, phrases, and verbs involving movement, not just frozen alphabet letters.
Facial expression grammar
Sign language grammar relies on facial expressions — raised eyebrows turn statements into questions. Adding computer vision facial analysis captures non-manual components the current system misses entirely.
NLP grammar-aware translation
ASL and ISL have syntactic structures that differ from spoken English. An NLP grammar correction layer would restructure transcribed sentences to match sign language syntax before animating — producing linguistically natural output.
TensorFlow Lite edge deployment
Model quantization for offline use on low-resource devices. Makes SignVoice viable in rural classrooms, low-resource clinics, and settings where reliable internet access cannot be assumed.
Mobile apps — Android and iOS
Extending beyond desktop web browsers dramatically increases reach, particularly in contexts where smartphones are the primary computing device.
AR sign overlays
Point camera at real-world objects and see sign language overlays contextually. Shifts from structured quiz learning to immersive, situational language acquisition.
